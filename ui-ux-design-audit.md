# UI/UX Design Audit & Polish

## When to activate
Activate when the user asks for a UI/UX audit, design review, or "make this site/app look/feel better" on a web or app frontend — including `/ui-ux-design-audit`. Also activate proactively partway through frontend work when you notice one of the defect patterns below, even if the user didn't ask for a formal audit.

## When NOT to use
Don't use this for pure backend/API work, for a single isolated bug fix the user has already diagnosed, or for greenfield design-system creation from scratch (that's a design task, not an audit/polish task — though the checklist below is still useful input to it).

## Instructions

### 1. Audit at two levels, not one
A UI/UX audit that only reads code misses things that only show up rendered (spacing, color clash, motion speed). An audit that only looks at a screenshot misses things baked into the implementation (dead click handlers, broken loops, oversized assets). Do both:
- **Code-level pass:** read the actual components, not just the design tokens. Click handlers that don't exist despite `cursor-pointer`. Routing that reloads the page instead of navigating client-side. Loop/animation math that only works for specific content lengths.
- **Rendered pass:** run the dev server and actually look at it (or ask the user to and describe what they see precisely — "grayish," "moves too fast," "empty spaces" are all real signal, triangulate the root cause from the phrasing).

When you can't visually inspect (no browser/screenshot tool available), say so explicitly rather than describing a screenshot you didn't take. A code-only audit is still valuable — just label it as one.

### 2. Checklist of defect patterns to specifically check for
These recur across nearly every frontend project and are easy to miss because each one looks "fine" until you know what to look for:

- **SPA navigation reloading the page.** If the app uses client-side routing (React Router, wouter, Next.js Link, etc.), grep for plain `<a href="/...">` tags pointing at internal routes — these force a full page reload, defeating the router. Hash-only anchors (`/#section`) are a legitimate exception; they need native browser scroll behavior, not client-side routing.
- **Scroll position not resetting on route change.** A direct consequence of fixing the above: client-side navigation does NOT reset scroll position like a real page load does. If you fix full-page-reloads, you must also add a scroll-to-top-on-route-change effect, or users land on the new page still scrolled to where they were on the old one. This is easy to forget as a paired fix.
- **Click-affordance mismatch.** Elements styled with `cursor-pointer` and hover states but no `onClick`/`href` — they look interactive and do nothing. Grep for `cursor-pointer` and verify every hit actually does something.
- **Marquee/carousel infinite-loop math.** A common "duplicate content 2x, translate -50%" seamless-loop pattern only works if a single copy of the content is wider than the viewport. Short lists (few items) go visibly blank mid-loop on wide screens. Fix: dynamically repeat the source array until it exceeds a safe minimum width/count before duplicating for the loop, don't hardcode the duplication factor.
- **Animation speed math breaking when content length changes.** If you fix the above by lengthening looped content, the existing animation-duration is now wrong (same duration, much more distance = feels sped up). Always recompute/re-tune duration whenever you change how much content is in a loop.
- **Pause-on-hover on a decorative ticker.** Feels like a good idea in isolation, reads as "stuck/broken" in practice for chrome elements (partner logos, service tags) that aren't meant to be read item-by-item. Keep those moving continuously; reserve pause-on-hover for content the user is meant to actually read (image carousels with captions, testimonials).
- **Dead/disabled UI elements shipped as visible.** Placeholder social icons, "coming soon" buttons with `cursor-not-allowed` — if there's no real destination yet, hiding the element usually reads better than showing it disabled. A visibly broken-looking control signals "unfinished" to visitors even if intentional.
- **Fake vs real logos/brand marks.** Icon-font/glyph libraries (Simple Icons, Font Awesome brand icons, etc.) give flat single-color stylized marks, not the actual official multi-color logo. For "trusted by X" / "we work with Y" partner strips, this is a real fidelity gap users notice immediately (a solid-red Ferrari glyph vs the real prancing-horse shield). Source real logos from the brand's own site or Wikimedia Commons/Wikidata's official logo property when accuracy matters, and say clearly which ones you couldn't verify — don't guess or fabricate.
- **Asset bloat causing perceived slowness.** A single oversized source file (an SVG with a multi-megabyte base64-embedded raster, an uncompressed 4K PNG used as a 60px icon) can dominate load time even when everything else is lean. When something "feels slow to load," check individual asset sizes before assuming it's a network/hosting problem — `ls -la` sorted by size finds this in seconds.
- **Color/visual-weight clustering in repeating lists.** Logo strips, tag clouds, and similar repeating collections built by simply appending items in the order you researched them tend to cluster same-color or same-tier items together (all the reds next to each other, all the exotic/loud brands bunched at one end). Deliberately interleave/round-robin across categories (or shuffle) instead of listing in discovery order.
- **Background/theme layering causing muddiness.** Dark themes especially: stacked semi-transparent surfaces (a `bg-card/50` section on a dark background, a translucent blurred nav, a global noise-texture overlay) each individually look reasonable in isolation but compound into a "grayish/washed out" feel. If a user says a dark theme "feels gray," check for the noise-overlay-style texture layer and the section-background opacity stacking before assuming the base color itself is wrong.
- **Icon set consistency.** Mixing icon styles (line icons here, filled emoji-style there, flat brand glyphs elsewhere) reads as unpolished even if each individual icon is fine. Pick one icon library/style and stay in it; if you need visual variety, vary color/treatment (badges, backgrounds) rather than mixing icon families.
- **Accessibility basics.** Skip-to-content link on heavy-motion pages, `aria-label` on icon-only buttons, honeypot fields properly hidden from real users but visible to bots, `prefers-reduced-motion` respected, focus-visible states not accidentally removed by custom styling.

### 3. Design philosophy for polish work
- **Motion should signal life, not decoration.** A static informational strip (services list, feature tags) benefits from a subtle continuous animation (icon micro-pulse, staggered breathing effect) that reads as "alive" without demanding attention. The failure mode in both directions: too static feels dead, too much/fast motion feels like a mockup or reads as sped-up/broken. When a user says something "doesn't feel alive," a small continuous ambient animation is usually the fix — not more static content.
- **Match visual language to brand tone, don't default to generic patterns.** A luxury/premium brand (dark theme, gold accents) reads wrong with bright multi-color SaaS-style icon badges — even if each individual choice (colored badges = "more professional/polished") is defensible in isolation, it can clash with the established tone. When making a stylistic call the user didn't specify, state your reasoning and flag it as a judgment call they may want to revisit, rather than presenting it as objectively correct.
- **Real content beats decorative filler.** Verified real logos, real business phone numbers, real stats > plausible-looking placeholders. If you can't verify something (a logo, a stat, a contact detail), say so explicitly rather than shipping a best-guess as if it were confirmed — especially for anything customer-facing on a live business site.
- **Iterate from precise user language.** Non-technical feedback ("it feels grayish," "it moves too fast," "not pleasing") is real signal with a findable root cause — don't dismiss it as vague. Translate it into the specific technical defect (opacity stacking, animation-duration math, color clustering) and fix that, rather than making an unrelated change and hoping it addresses the complaint.
- **Verify before/after every polish pass.** Typecheck, run the test suite, and — when a dev server is available — actually reload and look, before declaring a round of fixes done. Visual/UX changes are exactly the category of change where "it compiles" is not the same as "it looks right."

## Example
See the Bright Star Auto Ltd website audit and polish pass for a worked example of this full loop: an initial code-level audit surfaced dead gallery clicks and full-page-reload navigation; live user feedback then surfaced grainy background layering, a marquee loop that went blank on wide screens, flat single-color brand glyphs mistaken for real logos, an oversized embedded-raster SVG causing slow loads, color-clustered logo ordering, and an services strip that "didn't feel alive" — each traced to a specific defect from the checklist above and fixed individually, then verified with typecheck + tests + a live reload before moving on.
